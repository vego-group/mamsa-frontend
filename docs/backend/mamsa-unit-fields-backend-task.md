## 1. بيانات غلط ظاهرة للمستخدم الآن

### 1.1 `beds` — أهم بند في الملف

مورد الوحدة يرجّع `bedrooms` وما يرجّع `beds`. صفحة الوحدة تعرض أربع بطاقات: ضيوف / غرف نوم / **أسرّة** / حمامات. عشان ما تطلع فاضية، الـ adapter يعبّي الأسرّة من غرف النوم:

```ts
bedrooms: Number(u.bedrooms ?? 0),
beds:     Number(u.bedrooms ?? 0),   // ← نفس الحقل، معروض مرتين
```

النتيجة: كل وحدة في الموقع تعرض عدد أسرّة = عدد غرف النوم. شقة بغرفتين وأربع أسرّة تظهر «٢ أسرّة». الرقم مغلوط للضيف قبل الحجز.

**المطلوب:** `beds` (integer) على مورد الوحدة، ومدخل له في نموذج إضافة/تعديل الوحدة عند الشريك.

### 1.2 `owner.type` — كل مضيف يظهر «مالك فردي»

`owner` يرجّع `{ id, name }` بس. الواجهة تعرض سطر «مضيف موثّق · مالك فردي» تحت اسم المضيف، والقيمة **ثابتة في الكود**:

```ts
ownerType: 'individual',   // hardcoded — ما في مصدر لها
```

يعني الشركات تظهر كملّاك أفراد. عندكم التمييز أصلاً (فيه `is_partner` على المستخدم) بس ما يوصل على مستوى الوحدة.

**المطلوب:** `owner.type: "individual" | "company"`. ولو متاح `owner.avatar_url` (الواجهة حالياً ترسم أول حرف من الاسم في دائرة) و `owner.is_verified` عشان شارة «موثّق» تكون حقيقية مو ثابتة.

### 1.3 `features` — المرافق تحتاج مفتاح ثابت

`features` يرجّع مصفوفة **نصوص عربية**، والواجهة تطابقها بجدول عشان تختار الأيقونة:

```ts
const FEATURE_KEYS = { 'واي فاي': 'wifi', 'واي-فاي': 'wifi', مطبخ: 'kitchen', ... };
```

أي مرفق جديد، أو نفس المرفق بإملاء مختلف («تكييف» مقابل «مكيف»، مسافة مقابل شرطة) → يسقط للنص الخام بدون أيقونة. هش بالكامل ومربوط بالإملاء.

**المطلوب:** كائنات بدل نصوص:

```json
"features": [
  { "key": "wifi",    "label": "واي فاي" },
  { "key": "kitchen", "label": "مطبخ" }
]
```

`key` من قائمة مغلقة ثابتة عندكم (slug إنجليزي)، و`label` نص العرض. ابعثوا لنا القائمة الكاملة للمفاتيح ونجهّز أيقونة لكل واحد.

## 2. حقول الواجهة جاهزة لها والـ API ما يرجّعها

كلها موجودة في `src/types/index.ts` وفاضية دائماً.

### 2.1 الوحدة

| الحقل | الوضع الحالي | المطلوب |
|---|---|---|
| `taxPercent` | غير معبّأ | `tax_percent` — نسبة الضريبة للوحدة (لو تختلف عن ١٥٪ الموحّدة) |
| `isFeatured` | غير معبّأ | `is_featured` (bool) — قسم «وحدات مميزة» في الرئيسية |
| `hasDiscount` / `discountPercent` | غير معبّأ | `discount_percent` (int, nullable) — شارة الخصم على كرت الوحدة |
| `status` | يُشتق: `approved` وإلا `pending` | `approval_status` يرجّع الأربعة: `draft` \| `pending` \| `approved` \| `rejected` (+ `rejection_reason` لو مرفوضة) — لوحة الشريك تحتاج تفرّق |
| `country` | ثابت `"السعودية"` | `country` على المورد لما توسّعون خارج السعودية — مو مستعجل، بس سجّلوه |

ملاحظة عكسية: أنتم ترجّعون `area` (المساحة) والواجهة **ما تستخدمها**. لو تبونها تظهر في بطاقات الوحدة قولوا لنا ونضيفها.

### 2.2 المستخدم

| الحقل | الوضع الحالي | المطلوب |
|---|---|---|
| `firstName` / `lastName` | نقسم `name` على المسافات: `parts[0]` اسم أول، الباقي اسم أخير | `first_name` + `last_name` منفصلين. القسمة الحالية تكسر مع «عبد الله محمد» → الاسم الأول «عبد» |
| `avatarUrl` | غير معبّأ | `avatar_url` (nullable) على `GET /auth/me` و `GET /user/profile` |
| `role` | يُشتق من `is_admin` / `is_partner` | الشريك الفرد والشركة نفس القيمة حالياً — نحتاج `partner_type: "individual" \| "company"` (نفس بند §1.2) |

### 2.3 الحجز

| الحقل | الوضع الحالي | المطلوب |
|---|---|---|
| `guests` | `guests` رقم واحد → نحطه كله بالغين ونصفّر الأطفال | `guests: { adults, children }` — نموذج الحجز يجمعهم منفصلين ولازم يرجعون منفصلين |
| `guestName` | غير معبّأ | `guest_name` — اسم الضيف على الحجز (لوحة الشريك) |
| `userId` | ثابت `'CURRENT_USER'` | `user_id` على مورد الحجز |
| `review` | نفحص وجوده بس (`isReviewed`) والشكل غير محدد | ثبّتوا الشكل: `review: { id, rating, comment, created_at } \| null` |
| مراجعة → `userAvatarUrl` | غير معبّأ | `user_avatar_url` على مورد المراجعة |

## 3. اشتقاقات هشّة — تشتغل الآن وممكن تنكسر

### 3.1 `cancellation-preview` ما يرجّع إجمالي الحجز

الواجهة تعرض «المبلغ المسترد» و«المبلغ المخصوم». عندكم ترجّعون `refund_amount` و `refund_percent` بس، فنستنتج الإجمالي **بالقسمة**:

```ts
const total = refundPercent > 0 ? refundAmount / (refundPercent / 100) : refundAmount;
forfeitedAmount = total - refundAmount;
```

المشكلة: عند `refund_percent = 0` القسمة تنهار فنعتبر الإجمالي = صفر، والمبلغ المخصوم يطلع **صفر** بينما الصح هو كامل مبلغ الحجز. يعني إلغاء بدون استرداد يعرض «مخصوم: ٠ ر.س».

**المطلوب:** `total_amount` و `forfeited_amount` صريحين في رد `cancellation-preview`. ما نبي نحسب فلوس بالقسمة العكسية.

### 3.2 `tier_label` نص جاهز بدل بيانات

الرد يرجّع `tier_label` كنص مُصاغ فقط، فالواجهة تحط `tier: null` وتعرض النص كما هو — ما تقدر تبرز المستوى الحالي داخل جدول السياسة ولا تترجمه للإنجليزية.

**المطلوب:** مع النص، الحقول المنظّمة: `tier: { min_hours_before_checkin, refund_percent, label }`. نفس شكل `tiers[]` اللي ترجعونه في `cancellation_policy_details` — فقط العنصر المطابق.

### 3.3 `cancelled_by` — القيم المسموحة غير موثّقة

الواجهة تعمل cast مباشر للقيمة بدون جدول تحويل:

```ts
cancelledBy: b.cancellation.cancelled_by as 'customer' | 'partner' | 'admin' | 'system'
```

أي قيمة غير متوقعة تعدي بصمت وتكسر نص «أُلغي بواسطة…».

**المطلوب:** ثبّتوا القائمة كتابةً. لو تستخدمون أسماء ثانية (`guest` بدل `customer`, `host` بدل `partner`) قولوا لنا ونضيف جدول تحويل.

### 3.4 `cancellation_policy` القديم

`cancellation_policy` (enum نصي مثل `48_hours`) لسه يرجع، وإحنا نعتمد `cancellation_policy_details` كمصدر أساسي ونستخدم القديم fallback بس. لما تشيلونه نهائياً بلّغونا قبلها — الـ fallback عندنا يعتمد عليه للحجوزات غير المدفوعة اللي ما لها snapshot.
